# AGENTS.md

> 适用范围：本仓库内所有自动化 Agent 与人工协作者。
> 当前仓库事实优先，未来代码仓目标规范必须显式标注为 target state，不得写成已落地能力。

## 1. 仓库现状

- 当前仓库同时包含：`docs/` 方案文档、`src/` 参考实现代码、`tests/` 基础测试、`pyproject.toml` 构建清单、`deploy/` 部署示例与脚本。
- 当前可验证对象包括：文档内容一致性、接口与实现对齐、静态检查（例如 `python -m compileall`）、以及在具备依赖环境时可执行的测试与代码规范检查。
- 禁止虚构“测试已通过”“构建已通过”等结论；CI/门禁相关结论必须以可复现命令输出或实际 CI 运行结果为准。

## 2. 当前有效协作规则

### 2.1 文档优先

- 修改需求、架构、接口、目录、路线图中的任一项时，必须同步检查其余阶段文档是否仍然一致。
- 重大设计变更必须同时更新：
  - 范围与目标
  - 架构边界
  - 公共接口或目录约定
  - 验收标准与路线图
- 不允许只改一份文档而让其他文档继续保留旧结论。

### 2.2 术语与建模

- 运行时主对象统一使用 `tenant_id`、`session_id`、`knowledge_base_id`。
- `session` 用于承载一次可恢复的对话上下文；不要与 `conversation` 混用。
- 事件总线只能描述控制面或离线任务，不得描述实时语音热路径。
- 配置能力必须区分：
  - 启动时加载配置
  - 运行时动态配置
  不得把两者写成同一能力。

### 2.3 评审标准

- 优先识别会导致实现返工的问题：边界冲突、职责不清、性能口径缺失、安全模型缺失、路线图失控。
- 性能指标必须说明测量口径，例如 p50/p95、首包还是整句、稳态还是冷启动。
- 涉及企业、多租户、知识库上传时，必须覆盖最小安全模型：租户隔离、鉴权范围、文件限制、日志脱敏、外部模型数据边界。

### 2.4 操作边界

- 禁止为了“修正文档”而编造不存在的代码结构、命令输出或部署事实。
- 禁止破坏性操作，除非用户明确要求并可验证回滚路径。
- 禁止在文档中暴露真实密钥、Token、连接串或其他敏感信息。

## 3. 文档修改清单

开始前：

- 确认本次改动影响哪些阶段文档。
- 确认是否改变 MVP 范围、接口命名、目录结构或供应商边界。
- 先列出受影响术语：如 `session_id`、`tenant_id`、`RAG`、`RTC`、`事件总线`。

修改中：

- 每完成一个主题，立即做交叉搜索，确认旧表述已清理。
- 若新增约束，必须同步补充 Why，而不是只写结论。
- 文档中的示例、目录树、接口表、路线图必须相互对应。

结束前：

- 用搜索确认不存在已废弃术语或冲突表述。
- 汇总本次修正的设计问题、修改文件和剩余风险。
- 明确哪些内容是当前事实，哪些是 future target。

## 4. Future Target：代码仓工程规范

> 以下为未来代码落地时应遵守的目标规范；当前仓库尚未具备这些文件与命令。

### 4.1 目标工程结构

- 目标包结构：`src/customer_ai_runtime/`
- 目标构建清单：`pyproject.toml`
- 目标测试目录：`tests/unit`、`tests/integration`、`tests/contract`
- 目标部署目录：`deploy/` 或 `ops/`

### 4.2 目标质量门禁

- `python -m pytest`
- `python -m ruff check .`
- `python -m ruff format --check .`
- `python -m mypy src`

以上命令仅在代码仓落地后才可声明为实际门禁；在本仓库中只能作为目标约束，不得宣称已执行通过。

### 4.3 目标实现约束

- 统一使用 `tenant_id` 进行多租户边界隔离。
- 公共接口输入必须校验空值、类型、枚举、范围和权限范围。
- 日志默认结构化，禁止打印敏感字段原文。
- RAG 返回结果默认附带引用来源；若未命中知识库，必须显式走无检索分支。
- 实时语音链路采用媒体面与控制面分离设计，禁止把音频热路径建立在消息总线上。

## 5. 提交与变更说明

- Commit message 使用 Conventional Commits：`feat|fix|docs|refactor|test|chore|build|ci`
- 提交说明至少包含：
  - What：改了什么
  - Why：为什么改
  - How to verify：如何验证，当前文档仓可用“搜索/人工对齐检查”替代构建命令

## 6. 输出要求

- 默认使用中文。
- 优先给出高价值事实、风险和可验证结论，避免空泛总结。
- 发现仓库缺证据时，必须写“未发现”或“当前不可验证”，不能猜。
