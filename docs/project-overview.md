# 项目说明

## 1. 这个项目做了什么

本项目构建了一个智能客服平台参考实现，把文本客服、语音客服、RTC 实时语音通话、RAG 知识增强、业务系统联动和人工协同统一在一个模块化平台中。

## 2. 它解决了哪些真实业务问题

- FAQ 重复咨询占用人工
- 动态业务查询依赖多个后端系统
- 用户入口多而能力不统一
- AI 无法处理时缺少可靠转人工
- 知识库和策略缺少持续运营能力

## 3. 为什么单纯做 RAG 不够

因为客服问题不只有静态知识问答，还包含订单查询、投诉升级、账号验证、售后处理和实时语音通话。RAG 只能回答“知道什么”，不能独立完成“系统里现在是什么状态”和“该不该转人工”。

## 4. 为什么需要业务路由与工具调用

客服请求至少分为四类：

- 知识型：走 RAG
- 业务型：走工具或业务 API
- 人工型：直接转人工
- 风险型：优先升级处理

如果没有路由与工具调用，订单、物流、售后等动态问题会被错误地交给大模型猜测。

## 5. 为什么需要人工协同

复杂、高风险、情绪化或低置信度问题必须有人工兜底。平台通过会话摘要、意图提炼、历史消息和推荐回复，把 AI 与人工串成一个闭环，而不是相互割裂。

## 6. 为什么需要文本、语音、RTC 三种能力

- 文本适合大多数线上入口
- 语音适合移动端轻量咨询
- RTC 适合需要实时对话、连续打断和快速响应的场景

三者共用一个客服引擎，减少重复建设。

## 7. 为什么需要模块化设计

客服平台长期会替换模型、扩展渠道、接入新业务系统。模块化设计把渠道、引擎、提供商和运营管理分开，降低后续改造成本。

## 8. 为什么需要多厂商适配

企业真实环境会同时考虑成本、合规、时延和可用性。平台通过适配层允许替换 LLM、ASR、TTS、向量库和 RTC 提供商，避免被单一厂商绑定。

## 9. 各项技术分别承担什么职责

- FastAPI：HTTP 与 WebSocket 服务入口
- Session/Route/RAG/Orchestrator：核心客服引擎
- Provider Adapters：屏蔽外部厂商差异
- Runtime Config：支持运营动态调整 Prompt 与策略
- Metrics/Diagnostics：支撑观测与排障

## 10. 技术如何对应解决业务问题

- RAG 解决静态知识问答
- Tool / Business Adapter 解决动态业务查询
- Handoff Service 解决 AI 与人工交接
- Voice / RTC Service 解决语音与实时通话体验
- Admin API 解决运营、排障与策略调优

## 11. 当前项目已交付了什么

当前参考实现交付：

- 文档体系
- 可运行服务骨架
- 文本客服、知识库、业务工具、人工协同、语音、RTC、管理端接口
- 默认本地提供商
- 基于 `storage/state/` 的本地 JSON 持久化仓储
- 可选 OpenAI / Qdrant 适配器
- 可选 HTTP 业务系统适配器
- 可嵌入宿主系统的模块化接入入口
- 自动化测试与部署脚本

当前已实际验证：

- `.venv\Scripts\python.exe -m pytest`
- 结果：`10 passed`

当前未验证：

- Docker 构建运行
- OpenAI / Qdrant / 外部业务 HTTP 服务的真实联调

## 12. 如何接入、运行、维护

- 接入：调用 HTTP / WebSocket 接口，或参考 `examples/`
- 运行：使用 `.venv\Scripts\python.exe -m customer_ai_runtime`
- 维护：通过管理接口维护 Prompt、策略、知识库与诊断信息

## 13. 作为模块被其他系统接入

当前实现支持三种模块化接入方式：

1. 宿主系统通过 HTTP / WebSocket 调用独立运行的客服模块
2. 宿主 FastAPI 系统通过 `CustomerAIRuntimeModule.mount_to()` 挂载子应用
3. 宿主进程直接通过 `CustomerAIRuntimeModule.chat()`、`voice_turn()` 调用客服能力
