# 业务需求分析

## 1. 项目定义

本项目建设一个可复用、可集成、可扩展的智能客服平台，为企业提供统一的文本客服、语音客服、实时语音通话客服、知识增强问答、业务查询、人工协同与运营管理能力。

平台定位是“客服能力层”，而不是某个具体业务页面。业务系统通过 HTTP、WebSocket 或 SDK 协议接入平台，平台负责统一会话编排、知识检索、模型调用、工具查询、转人工和运营管控。

## 2. 真实业务问题

企业客服普遍存在以下问题：

1. 常见问题重复出现，人工客服被大量 FAQ 占用。
2. 业务查询问题需要跨订单、售后、物流、账号系统，处理慢且容易出错。
3. 用户触达入口多，文本、语音、APP、小程序、官网与电话链路割裂。
4. 单纯 FAQ 搜索或 RAG 命中率有限，无法处理动态业务状态。
5. 复杂问题、高风险投诉、情绪化用户必须转人工，但交接信息经常不完整。
6. 知识库、Prompt、模型策略缺少运营闭环，效果无法持续优化。
7. 多厂商能力耦合严重，切换模型、ASR、TTS、RTC 成本高。

## 3. 为什么不能只做简单 RAG

简单 RAG 只能覆盖“静态知识型问题”，无法独立解决以下场景：

- 订单、物流、售后等实时业务状态查询
- 用户明确要求人工或投诉升级
- 需要多轮上下文和状态流转的语音/通话会话
- 需要打断、超时、重试、回放与转接的 RTC 实时通话
- 需要运营人员动态调整策略、提示词和灰度配置

因此平台必须同时具备：

- 知识型问题的 RAG 检索
- 业务型问题的工具调用或业务 API 联动
- 高风险/低置信度的转人工策略
- 文本、语音、RTC 三种交互能力
- 运营管理与持续优化能力

## 4. 覆盖业务场景

### 4.1 文本客服

- 官网在线客服
- App 内客服
- 小程序 / H5 客服
- FAQ / 帮助中心智能问答
- 工单前置分流

### 4.2 语音客服

- App 内语音咨询
- 语音输入、语音播报回复
- 非通话型语音交互

### 4.3 实时语音通话客服

- 基于 RTC 建立房间并接入双向语音
- ASR 实时识别用户语音
- RAG / 工具 / LLM 生成回复
- TTS 合成语音并回推
- 支持打断、静默超时、重试、转人工、通话结束

### 4.4 AI 与人工协同

- 低置信度问题转人工
- 投诉、退款纠纷、敏感问题转人工
- 用户主动要求人工
- AI 生成会话摘要、意图、上下文和推荐回复供人工接手

### 4.5 业务查询与处理

- 订单状态查询
- 售后进度查询
- 物流轨迹查询
- 账号相关查询
- 业务系统动作触发与结果回传

### 4.6 运营与管理

- 知识库导入与更新
- Prompt 模板管理
- 模型/路由/阈值配置
- 会话查询与故障排查
- 指标分析与策略灰度

## 5. 角色与使用方式

### 5.1 最终用户

- 通过 Web、App、H5、小程序发起文本或语音咨询
- 在复杂问题下选择转人工
- 在 RTC 房间中完成实时语音通话

### 5.2 一线人工客服

- 查看 AI 交接摘要、意图、历史消息、推荐话术
- 接管复杂或高风险会话

### 5.3 运营人员

- 维护知识库、Prompt、分类策略和阈值
- 观察命中率、转人工率、响应时延与失败率

### 5.4 平台研发 / 集成人员

- 通过 API / SDK / WebSocket 接入自身业务系统
- 替换 LLM/ASR/TTS/RTC/向量库/业务 API 提供商

## 6. 核心业务流程

### 6.1 文本流程

1. 渠道接入层接收文本消息并做租户鉴权。
2. 会话层恢复 `session_id` 上下文。
3. 路由策略识别知识型、业务型、人工型或高风险。
4. 知识型走 RAG，业务型走工具或业务 API。
5. 编排层融合上下文、检索结果和工具结果生成回复。
6. 若低置信度或高风险则进入人工协同。
7. 返回文本回复、引用来源、路由结果和下一步动作。

### 6.2 语音流程

1. 接收语音音频或音频片段。
2. ASR 识别文本并附带识别置信度。
3. 进入统一会话和路由流程。
4. 生成文本回复后交给 TTS 合成音频。
5. 返回文本结果、音频结果、路由结果和状态信息。

### 6.3 RTC 通话流程

1. 创建房间并建立 WebSocket 控制与媒体事件通道。
2. 用户发送音频分片，房间状态进入 `listening`。
3. ASR 输出增量文本，策略层判断是否需要打断或继续等待。
4. 编排层生成答复，TTS 输出音频片段。
5. 服务端通过 RTC 事件协议回推文本、音频和状态事件。
6. 发生静默超时、投诉、低置信度、人工请求时切换状态并触发转接。

## 7. 成功指标

业务指标必须明确测量口径：

- AI 首轮解决率：用户首轮问题无需人工即可结束会话的比例
- 总体解决率：含 AI 与人工协同后的闭环解决比例
- 转人工率：全部会话中进入人工协同的比例
- 知识命中率：知识型问题中成功检索有效引用的比例
- 业务工具成功率：业务查询调用成功返回的比例
- 文本响应时延：从接收请求到首个响应返回的 p50 / p95
- 语音首包时延：从接收音频到首个文本结果或音频结果返回的 p50 / p95
- RTC 通话轮次成功率：通话中每一轮问答未超时、未失败的比例
- 客诉升级率：投诉类会话中升级至人工的比例

## 8. 最小安全模型

本项目涉及用户数据、租户边界与业务查询，必须至少覆盖以下威胁模型：

1. 多租户越权：任意请求都必须绑定 `tenant_id` 并校验 API Key 的租户范围，防止跨租户读取会话、知识库和业务数据。
2. 敏感信息泄露：日志、错误信息、诊断事件默认脱敏，不输出明文 API Key、账号标识、完整业务数据。
3. 外部模型边界：发送给外部 LLM/ASR/TTS 的数据必须是最小必要集，知识库与业务数据要受策略开关控制。

## 9. 非功能要求

- 架构解耦：渠道层、引擎层、提供商层、运营层职责分离
- 可替换：LLM/ASR/TTS/RTC/向量库/业务 API 可通过适配层切换
- 可维护：配置、提示词、路由阈值与知识库可独立管理
- 可观测：记录请求日志、错误日志、指标和会话诊断事件
- 可测试：至少包含单元测试、集成测试、关键链路测试
